# agent-glue Review for First-Pass Dialog System Port

Reviewer: dialog-domain-a sub-agent. Reviewed against the SC first-pass animation system (`.claude/skills/dialog-domain/scripts/first-pass/`, `GameConfigs/Tools/FirstPass/`).

## 1. Strong fit

**Disposition + variant-dispatch edges map cleanly onto the per-line outcomes the first-pass selector already produces.** Today `selector.py` returns picks that are either accepted, accepted-but-out-of-pool (currently a retry), or rejected (logged and skipped). An iron `Disposition = Accepted | AcceptedWithAudit | Rejected` per line, dispatched without predicates, is exactly the shape -- the "AcceptedWithAudit" variant has an obvious home for the "LLM picked X, but disable rule downgraded to Y" cases that today are buried in selector logs. The validate-retry sub-graph pattern (propose -> validate -> NeedsRetry back to propose) is precisely how `validate_assignment` already wants to behave but doesn't cleanly express.

**The show-your-work-as-cache mechanism is a near-perfect generalization of `inputs_hash`.** First-pass already keys staleness on a per-line hash of (line text, direction, pool, prompt, disables). WorkRecord-keyed-by-request-hash with `InvalidationCriteria` is the same idea, just one level up at the LLM-call boundary. The "identical request -> cached result without re-invoking the worker" semantic is exactly what `generate --all` is trying to achieve via prompt-cache warming today -- and unlike prompt-cache warming, the agent-glue cache survives process restarts and is replayable in cohort mode. **This is the single biggest win.** First-pass currently re-walks every conversation on `generate --all` and depends on Anthropic's server-side prompt cache to keep cost down; an agent-glue port gets disk-resident, hash-keyed cache that hits even after a restart or on a peer machine.

**Cohort substrate solves "regression tests for first-pass" cleanly.** Today there's no good way to lock in "the assignments Maya gets for `quest_amano_friendship_amanoArticle_Start` given prompt v7 + tags v3" as a regression test. Hand-authored or harvested WorkRecord cohorts give exactly that, and `replay --strict` failing when a recording is missing is the right default for CI.

**Canonical artifacts with `format: managed` map onto briefs.** Per-conversation briefs preserve hand-edited fields (personality, conversation_direction overrides) across regens. `format: managed` -- "the node owns the write" -- is exactly the contract `persist_brief` needs. The fact that the example pipeline (`character_animation`) was clearly designed with first-pass in mind shows: every node in the example corresponds 1:1 with a real first-pass step.

**Sidecar conventions (prompt.md, rules.yaml) map onto existing hand-authored config.** `prompts.yaml`, `disables.yaml`, `tags.yaml` all become rules-sidecars on the appropriate nodes; the Jebby/Justin prompt becomes the `propose_assignments` prompt.md. No new format to learn for designers.

## 2. Partial fit, doable

**Per-line `inputs_hash` vs request-level `request_hash`.** agent-glue's cache key is the whole WorkRequest. First-pass needs sub-request granularity: for a conversation with 20 lines where one line's text changed, you want to re-invoke the LLM for that one line, not all 20. The graph-design answer is "enumerate stale lines as a node, pass the stale subset to `propose_assignments` as a single call" -- which is exactly the example A shape. Workable, but the cache key for `propose_assignments` is now "the stale-subset prompt," not "the conversation," so cache hits across runs require the *same exact* stale-subset to recur. That's fine for "re-run same conversation, nothing changed -> full cache hit," but a designer who edits one line on a 20-line conversation will produce a 1-line request whose hash has never been seen before. Acceptable, just a different cache-hit shape than today.

**Per-conversation brief = both input and output.** Briefs are read at the top of the pipeline (load prior direction + per-line `assignment_claude`) and written at the bottom (merge new picks). agent-glue handles this via `load_brief` (in-impl read) + `persist_brief` (`format: managed`). The wrinkle: the brief is the *input* to staleness checking but also the *cache invalidator*. Putting brief mtime in `CacheControl.invalidate_if` would force a re-run every time the brief is rewritten, which is wrong. The right shape is "hash the brief-derived inputs (direction, prior picks, prompt, pool, disables) into the WorkRequest itself" -- the request_hash naturally invalidates when any of those changes, and the brief mtime is irrelevant. Doable, but consumers have to be disciplined: anything that should invalidate the cache must be IN the request, not adjacent to it.

**Prompt-cache-warm sequential ordering of `generate --all`.** The DESIGN explicitly calls this out as a v1 non-goal: "Sequential-ordered fan-out... stay as Python driver loops that call `runtime.run` per item." For first-pass, the bulk runner becomes a Python driver loop that calls the graph once per conversation, grouped by primary speaker. This is the same shape as today, just lifted one level. **No regression**, but no graph-system help either -- the speaker-grouping discipline stays in the driver.

**The "two phases for prompt-cache reuse" trick.** `generate --all` today does direction-then-assignments in two sweeps so the system prompt + character pool stays warm across calls. With the disk cache, the *cost* argument for that ordering weakens (a cold re-run on identical input is now free), but the *latency* argument (per-call API roundtrip) doesn't. Workable as a driver-loop concern, not a kit concern.

## 3. Genuine mismatch

**p4 edit / file mutation outside the cache contract.** `apply` opens the dialog CSV for edit in Perforce and rewrites `[FIRST PASS]` rows. agent-glue's caching semantics assume the recorded result *describes* the output of a deterministic transformation. A `claude_agent` worker that runs `p4 edit` and writes to a CSV will record "I did the p4 edit" -- but on cache hit, no p4 edit actually happens. The brief is the durable artifact; the CSV write is a side effect that must always run if the CSV's current state doesn't match the brief. The fix is to keep CSV writes in *in-impl* nodes (no caching), reserving `submit()` for LLM work. The DESIGN acknowledges this ("if it's a cheap data-shape transform, do it in-impl") -- but the line between "in-impl" and "delegate" has to be drawn carefully, and CSV writes that need p4 edit are emphatically in-impl.

**"Designer intent" mapping isn't a pipeline shape.** First-pass exposes verbs (`generate`, `apply`, `revert`, `report`) that compose with scope axes (identity_scope x generation_state_scope) and produce canonical phrasings that get spoken back to the designer. None of that is graph-runtime work; it's CLI surface area and a skill. agent-glue's `bin/agent-glue run/replay/promote-fixture` CLI doesn't help here. **Not a mismatch agent-glue should fix** -- the dialog-domain skill stays on top, and agent-glue is just the engine underneath -- but worth noting that porting first-pass means porting *the pipeline*, not the entire designer-facing system.

**Single-speaker hard-error and other "structural rejection" gates.** `generate_conversation` hard-errors on multi-speaker conversations *before* any LLM call. In agent-glue this is a node that returns `Rejected("multi_speaker", metadata={speakers: [...]})` and routes to a terminal. Doable, but the iron-contract "every input gets a recorded disposition" interacts oddly with "this conversation is structurally ineligible and we want a loud error, not a silent terminal." The hard-error today is more visible to the designer than a Rejected terminal would be. Minor accommodation, but worth noting.

**Report / inspect commands have no clean graph shape.** `report --rollup`, `report --top-impact`, `report --invalidated-by` are corpus-wide queries that read every brief + every CSV, join, and aggregate. These aren't pipelines; they're queries over the artifact corpus that the pipelines produce. They'd remain pure Python scripts; agent-glue offers them nothing. Not a mismatch per se, but a reminder that "port first-pass to agent-glue" really means "port the generate/apply graph, leave inspect/revert as scripts."

## 4. Missing primitives

- **Per-element invalidation in a list-shaped input.** First-pass needs "compute per-line `inputs_hash`, pass only the stale subset to one LLM call." The example pipeline calls this out as "subset-as-input-to-one-call, NOT fan-out." Workable today (enumerate-stale node + pass subset), but there's no kit help for the per-element hash computation. **Suggestion:** a `work-system` helper -- something like `compute_subhashes(items, key_fn)` that the consumer can use to construct cache invalidation criteria at a finer grain than the whole request. Not a hard requirement; first-pass already has this code in `inputs_hash.py`. But if multiple consumers reinvent it, it's a kit-level utility.

- **Speaker-grouped bulk runner with sequential ordering.** The non-goal explicitly punts this to the consumer. First-pass would re-implement the speaker-grouping driver loop in Python. **Suggestion (post-v1, work-system):** a `submit_batch(requests, group_key=..., parallel_per_group=False)` that handles the "same group runs sequentially in one process to keep prompt cache warm" pattern. Useful for any LLM-heavy pipeline with grouping keys; first-pass is just one of them.

- **"Validate against external pool" as a rules-yaml primitive.** First-pass's `validate_assignment` checks each pick against the speaker's allowed-animation set, which is derived from `npcs.yaml` + `animation-sets.yaml` + `disables.yaml`. The example pipeline gestures at this with `rules.yaml`, but real first-pass needs the pool resolved per-speaker dynamically. **Suggestion (graph-system, but really just a documentation/example matter):** show a node where `rules.yaml` is a *registry* and the node looks up the relevant rule subset from the input. Not a new primitive, but the example A as currently sketched understates how the rules sidecar actually gets used.

- **Sub-process isolation for p4 / shell side effects.** First-pass's `apply` does `p4 edit` then file writes. agent-glue has no opinion on side effects beyond "the worker can populate `side_effects` in WorkResult." For the in-impl path this is irrelevant. **Not actually missing** -- just confirming the kit deliberately doesn't try to manage external side effects, and first-pass would handle p4 the same way it does today.

- **Glossary / state-term enforcement.** Dialog-domain skill enforces a canonical vocabulary (`generated`, `not_generated`, `fresh`, `stale`, `in_sync`, `out_of_sync`). No agent-glue analog -- but this is correctly outside agent-glue's scope (it's a skill concern, not a pipeline concern). Listing here only to note explicitly that the skill stays on top.

## 5. Bottom line

**Porting first-pass to agent-glue would simplify the pipeline meaningfully, especially the cache+audit story, but it would not collapse the overall system.** The big win is replacing first-pass's bespoke `inputs_hash` plumbing and prompt-cache-warm bulk-runner discipline with a kit-level WorkRecord cache that hits across processes, supports cohort replay, and gives free regression-test infrastructure. The validate-retry sub-graph pattern is also a clean rewrite of selector retry logic. The graph topology of generate (`load -> check freshness -> propose direction -> enumerate stale -> propose assignments -> validate -> persist`) is a faithful and lossless re-expression of what the code does today, and the DESIGN's example A was clearly written with this port in mind.

What does *not* simplify: the designer-facing CLI verbs (`generate`/`apply`/`revert`/`report`), the scope-axis grammar, the dialog-domain skill, the p4 integration, the speaker-grouping bulk-runner loop, the inspect/report commands, and the single-speaker hard-error. Those stay as Python on top of the graph. Net effect: ~half of `firstpass_ops/` goes away (the pipeline-internals half) and ~half stays (the orchestration, scope, and reporting half).

**Single biggest open question before committing to the port:** *How does the WorkRecord cache invariant survive a brief edit that's not supposed to invalidate?* Designers regularly hand-edit `personality` on character files and per-conversation `direction` overrides on briefs. The first should not invalidate cached assignments; the second should. agent-glue's cache key is the WorkRequest yaml content -- the porter has to be deliberate about exactly which brief fields enter the request (and thus the hash) and which are read by the graph but excluded from the hash. Get this wrong in either direction and either (a) every personality tweak forces full corpus regen, or (b) direction edits silently do nothing. The current first-pass `inputs_hash` is hand-tuned for this; the port has to preserve that tuning explicitly in the request shape.
