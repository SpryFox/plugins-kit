# agent-glue: Architecture The cross-subsystem interfaces and the dependency graph. Shared architectural patterns live in `core/ARCHITECTURE.md.html`; subsystem-specific patterns live in each subsystem's ARCHITECTURE.md.html. **Visual companion:** [architecture.html](architecture.html) -- interactive dependency-graph diagram with dark/light toggle and image export. Same content as the ASCII graph below; the HTML version distinguishes v1-active subsystems from the deferred claude-work-queue + claude-workers slot visually. ## Subsystem dependency graph ``` core | +-- claude-work-queue | +-- work-system ---- (claude_inference, claude_agent worker submitters depend on claude-work-queue) | +-- graph-system --- (depends on work-system at the submit() integration point) ``` The DAG is strict; no cycles. Within work-system, the `openrouter` and `python_script` workers have no claude-work-queue dependency; only the two claude-backed workers do. Within graph-system, the runtime depends on the work-system's `submit()` once nodes start delegating; pure in-impl nodes need only core. ## Cross-subsystem interfaces ### graph-system to work-system The graph subsystem reaches the work subsystem through one function: ```python from agent_glue_lib.work import submit, WorkRequest result = submit(request=..., cohort=..., bypass_cache=...) ``` The graph subsystem constructs `WorkRequest` instances and calls `submit()`. The work subsystem fulfills them and returns `WorkResult`. Swapping worker types on a per-call basis is a request-config change; the caller's code is unchanged. The graph subsystem does not import any other names from `agent_glue_lib.work`. `submit()` is the entire surface. ### work-system to claude-work-queue The work subsystem's `claude_inference` and `claude_agent` worker submitters dispatch through the claude-work-queue's consumer API: ```python from agent_glue_lib.claude_work_queue import submit, wait item_id = submit(prompt=..., input=..., schema=...) output = wait(item_id, timeout=...) ``` The other two workers (`openrouter`, `python_script`) do not import from claude-work-queue. ### Any subsystem to core Any subsystem may import: - The loader, validator, and catalog types from `agent_glue_lib.core`. - Cross-cutting components from `core/components/` (by name in entity-type yamls). - The Disposition primitive from `agent_glue_lib.core.disposition`. Core does not import from any other subsystem. ## Cohort directory convention Cohorts (per-graph collections of replay fixtures) live under the graph that owns them. The work subsystem's recordings directory is a sibling within the same cohort: ``` graphs//cohorts// cohort.yaml # Cohort entity inputs/.yaml # Fixture entities expected/.yaml # ExpectedOutcome entities recordings//.yaml # WorkRecords (cohort-mode submit() looks here) ``` Cohort recordings reuse the work subsystem's record format and lookup mechanism -- cohort-mode at `submit()` time just points the cache directory at the cohort's `recordings/`. No separate "cohort recording" entity; recordings are WorkRecords in a different home. This is the single concrete coupling between the graph subsystem's cohort substrate and the work subsystem's cache substrate.
